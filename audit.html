Projektportrait

shadow-objects ist ein experimentelles, reaktives Entity-Component-Framework (ECS) für Web-UI-State-Verwaltung. Die zentrale Idee: Anwendungslogik in einen Shadow Environment auslagern, der wahlweise im Hauptthread (LocalShadowObjectEnv) oder in einem Web Worker (RemoteWorkerEnv) läuft. Custom Elements (<shae-ent>, <shae-prop>, <shae-worker>) verknüpfen das DOM mit Entities und Shadow Objects (Komponenten) über String-Tokens. Reaktivität liegt bei @spearwolf/signalize (Signals/Effects) und @spearwolf/eventize (Event-Emitter), beide aus der Hand desselben Autors. Aktuell ist das Framework als „highly experimental“ markiert; das Build-System wurde im Mai 2026 grundlegend erneuert (esbuild + turbo + biome).

   Datenfluss
   ──────────
   Downstream (Props):  View → Kernel → Entity → Shadow-Object-Signal
   Upstream  (Events):  Shadow-Object → Entity → Kernel → View
   Lateral   (Context): Eltern → Kind via SignalsPath  (Provider/Consumer)

View- und Worker-Seite sind Spiegelbilder, verbunden durch das asynchrone IShadowObjectEnvProxy-Protokoll. Im lokalen Modus läuft der Kernel im Hauptthread; der gesamte postMessage-Pfad entfällt, der ChangeTrail wird per structuredClone kopiert.

Executive Summary

Das Framework ist konzeptionell sauber und kompakt: Eine schmale öffentliche API, klare Schichtung (View ↔ Worker als Spiegel über ein 4-Methoden-Proxy), gute Testabdeckung des ECS-Herzens (Kernel, Entity, Registry, ShadowObjectCreationAPI). Reaktivität ist konsequent auf eigene Libraries gestützt; Build-System und Tooling sind nach dem Mai-2026-Renewal modern und in sich konsistent.

Die Risiken liegen außerhalb des ECS-Kerns — auf der View-/Worker-Brücke und in den Element-Lifecycle-Pfaden: Drei high-Severity-Probleme (kaputtes Quick-Start-Beispiel im publizierten README, fehlende E2E-Verifikation in CI, kein error/messageerror-Handling im Worker) sollten vor 1.0 adressiert sein. Hinzu kommen 14 medium-Befunde, davon mehrere zur asynchronen Symmetrie zwischen LocalShadowObjectEnv und RemoteWorkerEnv: das waitForConfirmation-Flag wird lokal ignoriert, die envProxy-Reassign-Race ist offen, syncWait() hängt nach destroy() für immer. strictNullChecks: false und Default-strukturierter-Clone im Local-Env sind ebenfalls echte Engagements für die Zukunft.

Dieser Re-Audit fügt der vorherigen Iteration (gleichentags) zwölf vertiefte Code-Findings hinzu; keines davon ist Regression — alle waren schon im Code, wurden aber im flacheren Erst-Scan nicht ausgewiesen.

Offene Fragen / Unklarheiten

  • Auto-Destruction-Re-Parenting unter Worker-Race-Bedingungen — Die 2026-05-09-Backlog-Auflösung (KERN-1/2/3) ist im Local-Pfad und im Playwright-E2E geprüft. Was passiert in einem RemoteWorkerEnv, wenn ein Property-Update und ein SetParent im selben ChangeTrail unterschiedlich von autoDestructionOnParentRemoval abhängen? Aktuell kein direkter Test.
  • Migrationsstand der Top-Level sideEffects-Liste — Die Liste enthält noch build/src/...-Pfade aus der alten Build-Pipeline. War das Absicht (Übergangsfenster) oder vergessen? Wenn vergessen → SIDE-EFFECTS-001 direkt fixen.
  • Worker-Reload-Pfad — WORKER-001 empfiehlt einen WorkerFailed-Event. Wer übernimmt die Recovery: ShadowEnv-Consumer (App-Layer) oder die Lib selbst (Auto-Reconnect)? Eine Entwurfsentscheidung, die das öffentliche API formt.
  • Verbleib der compare-Cache-Hit-Semantik — KERN-7 im 2026-05-09-Backlog wurde mit Warn-Verhalten gefixt: bei Cache-Hit mit abweichendem compare wird console.warn emittiert, aber der alte Reader bleibt. Ist das langfristig die richtige Semantik oder nur ein Kompatibilitäts-Übergang?
  • Performance-Default für Local-Env — siehe PERF-CLONE: ist structuredClone bewusst der Default, um Local und Remote semantisch identisch zu halten (kein versehentliches Sharen), oder ist das ein historischer Default, der gekippt werden sollte?

Optimierungspotenzial

Bewusst getrennt von Bugs/Findings: Verbesserungen, die kein Defekt sind.

  • API-Aufräumen vor 1.0: onDestroy-Tripel-Bedeutung (Symbol / Event / Callback) dokumentieren oder trennen; appendRoute aufteilen (Alias vs. @-Prop-Routen); IShadowObjectEnvProxy.isDestroyed ist da, aber ein ready-Promise und ein error-Event fehlen weiterhin.
  • RAF-Coalescing als optionaler Modus: bei High-Frequency-Properties (Animationen, Pointer-Move) sind heute n Updates = n postMessage. Eine optionale Sammelflanke pro Frame würde Worker-Roundtrips drastisch reduzieren.
  • Transferable-API auch für ChangeProperties: aktuell werden ArrayBuffer-Properties strukturell kopiert, nur SendEvents unterstützt transferables.
  • SignalsPath inkrementelles Subscribe: heute wird der Effekt bei jedem Add/Remove zerstört + neu erzeugt — bei dynamischen Provider/Consumer-Hierarchien teuer.
  • Entity.addChild Sort-Strategie: einmal pro Batch sortieren statt pro Insert würde Batch-Inserts O(n²) → O(n log n) reduzieren (siehe ENT-SORT).
  • peerDependencies für @spearwolf/eventize/signalize deklarieren — beide sind Identitätsempfindlich (Symbols, Module-Singletons). Bei Mehrfach-Resolutionen entstehen Duplikate, die Reaktivität lautlos brechen.
  • Doctest-Smoketest für README-Snippets — würde DOC-001 langfristig verhindern.
  • Vendor-doc für globalThis-Singletons — siehe GLOBAL-SINGLETON; SSR- und Multi-Tenant-Fragen sind heute nur im Code spürbar.
